# 開発ガイド

## ブランチ運用

- `main` では直接作業しません。
- `main` は常に安定した状態を保ちます。
- 作業は可能な限り GitHub Issue を起点にします。
- 各タスクごとに `main` から作業ブランチを切ります。
- 個人開発であっても、完了した変更は Pull Request を通して `main` に取り込みます。

推奨するブランチ名:

- `feature/<issue-number>-<short-description>`
- `fix/<issue-number>-<short-description>`
- `chore/<issue-number>-<short-description>`

例:

- `feature/12-add-minimap-overlay`
- `fix/34-correct-opacity-sync`
- `chore/56-update-dev-config`

## 標準的な作業手順

1. GitHub Issue を作成する、または既存の Issue を選ぶ。
2. ローカルの `main` を `origin/main` と同期する。
3. `main` から新しい作業ブランチを作成する。
4. 変更はその作業ブランチ上でのみ行う。
5. レビューしやすい粒度でコミットする。
6. `main` 向けの Pull Request を作成する。
7. 内容を確認して問題なければマージする。

## このリポジトリでのルール

- `main` は統合済みの変更だけを置くブランチとします。
- `main` への直接コミットは避けます。
- 必要に応じて `origin/main` の変更を作業ブランチへ取り込み、差分を最新に保ちます。
- 作業範囲が当初の想定より大きくなった場合は、別の Issue とブランチに分けます。
- 1つのブランチでは、できるだけ1つの論理的な変更に集中します。

## ベストプラクティスとの整合性

この運用は一般的なベストプラクティスと整合しており、特に不自然なやり方ではありません。

個人開発においては、「必ず GitHub Issue を作る」という点だけはやや厳密寄りです。ただし、各ブランチの目的が明確になり、後から履歴を追いやすくなるため、運用としては十分合理的です。非常に小さな修正では少し重く感じることもありますが、履歴を整理したいのであれば良い方針です。

## Codex の作業方針

Codex がこのリポジトリで作業する場合、以下の方針に従います。

- 明示的な指示がない限り、`main` で直接変更しません。
- 可能な限り Issue に紐づくブランチから作業を始めます。
- ブランチ名には Issue 番号を含めることを優先します。
- `main` へ戻すときは Pull Request を基本ルートとします。
