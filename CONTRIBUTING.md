# 開発ガイド

## ブランチ運用

- `main` では直接作業しません。
- `main` は常に安定した状態を保ちます。
- 作業は可能な限り GitHub Issue を起点にします。
- 各タスクごとに `main` から作業ブランチを切ります。
- 個人開発であっても、完了した変更は Pull Request を通して `main` に取り込みます。

推奨するブランチ名:

- `feature/<issue-number>-<short-description>`
- `fix/<issue-number>-<short-description>`
- `chore/<issue-number>-<short-description>`

例:

- `feature/12-add-minimap-overlay`
- `fix/34-correct-opacity-sync`
- `chore/56-update-dev-config`

## 標準的な作業手順

1. GitHub Issue を作成する、または既存の Issue を選ぶ。
2. ローカルの `main` を `origin/main` と同期する。
3. `main` から新しい作業ブランチを作成する。
4. 変更はその作業ブランチ上でのみ行う。
5. レビューしやすい粒度でコミットする。
6. `main` 向けの Pull Request を作成する。
7. 内容を確認して問題なければマージする。

## このリポジトリでのルール

- `main` は統合済みの変更だけを置くブランチとします。
- `main` への直接コミットは避けます。
- 必要に応じて `origin/main` の変更を作業ブランチへ取り込み、差分を最新に保ちます。
- 作業範囲が当初の想定より大きくなった場合は、別の Issue とブランチに分けます。
- 1つのブランチでは、できるだけ1つの論理的な変更に集中します。

## ベストプラクティスとの整合性

この運用は一般的なベストプラクティスと整合しており、特に不自然なやり方ではありません。

個人開発においては、「必ず GitHub Issue を作る」という点だけはやや厳密寄りです。ただし、各ブランチの目的が明確になり、後から履歴を追いやすくなるため、運用としては十分合理的です。非常に小さな修正では少し重く感じることもありますが、履歴を整理したいのであれば良い方針です。

## Codex の作業方針

Codex がこのリポジトリで作業する場合、以下の方針に従います。

- 明示的な指示がない限り、`main` で直接変更しません。
- 可能な限り Issue に紐づくブランチから作業を始めます。
- ブランチ名には Issue 番号を含めることを優先します。
- `main` へ戻すときは Pull Request を基本ルートとします。
- コミット、push、Pull Request 作成、レビュー返信の前に `AGENTS.md` とこのファイルの運用ルールを確認します。

## コミットメッセージ

- コミットメッセージは日本語で書きます。
- 1コミットは、できるだけ1つの論理的な変更に集中します。
- タイトルは `type: 日本語の要約` の形式にします。
- 必要な場合は `type(scope): 日本語の要約` の形式にしてもよいです。
- `type` は原則として `feat` / `fix` / `docs` / `test` / `refactor` / `chore` / `build` / `ci` / `perf` から選びます。
- タイトルの要約部分は短い名詞句または動詞句にします。
- タイトルに `[codex]`、`Codex:`、`Generated by Codex` などの実行者名・生成元表記は入れません。
- タイトル末尾に句点は付けません。
- 補足が必要な場合だけ、空行を挟んで本文に理由、影響範囲、確認内容を書きます。

例:

- `feat: 複数サンプル画像の認識検証を追加`
- `fix: 検証用デバッグ出力の衝突を防止`
- `docs: PR とコミットの運用ルールを追加`
- `feat(overlay-ui): overlay DTO に個体差分表示を追加`

## Pull Request

- Pull Request のタイトルと本文は日本語で書きます。
- タイトルに `[codex]`、`Codex:`、`Generated by Codex` などの実行者名・生成元表記は入れません。
- タイトルは変更内容を短く説明します。
- 本文は必要十分な情報に絞り、レビューしやすい粒度で書きます。
- ローカル環境の絶対パスなど、レビューに不要な個人環境情報は書きません。

本文には原則として以下を含めます。

- 概要
- 主な変更点
- 確認内容
- スコープ外にしたこと、または触っていない領域

例:

````md
## 概要

vision-py に複数サンプル画像をまとめて検証する導線を追加しました。

## 主な変更点

- `assets/samples/` 配下の画像を一括処理
- 画像ごとの debug 出力と validation report を保存
- 画像単位で success / partial / failed を集計

## 確認内容

```sh
PYTHONPATH=vision-py/src python3 -m unittest discover -s vision-py/tests -p 'test_*.py' -v
git diff --check
```

## スコープ外

- OCR ロジックの大改修
- Go 側と overlay-ui の変更
````

## レビュー返信

- レビュー指摘に返信する場合は、対応内容と確認内容を簡潔に書きます。
- 再レビューを依頼する場合は、最後に `@codex 再レビューお願いします` を付けます。
- 返信にも、レビューに不要なローカル絶対パスは書きません。
